def deco(f):
    cache = {}
    def wrapper(n):
        if n not in cache:
            cache[n] = f(n)
        return cache[n]
    return wrapper

@deco
def fib(n):
    if n < 2: return n
    return fib(n-1) + fib(n-2)

📌 C언어 연결 리스트(Linked List)와 포인터 심층 개념

연결 리스트(Linked List)는 각 노드(Node)가 데이터와 다음 노드를 가리키는 포인터(Pointer)로 구성된 선형 자료구조입니다. 배열과 달리 메모리가 연속적으로 할당되지 않아 삽입·삭제가 O(1)로 빠르지만, 임의 접근(Random Access)은 O(n)이 소요됩니다. 재귀 함수(Recursion)와 피보나치 수열 - 함정 포인트까지

재귀 함수(Recursive Function)는 자기 자신을 호출하는 함수로, 반드시 종료 조건인 기저 조건(Base Case)이 필요합니다. 피보나치 수열은 재귀의 대표 예시로, f(n) = f(n-1) + f(n-2)의 점화식을 따릅니다.

정보처리기사 실기에서 가장 많이 틀리는 함정은 f(0)의 반환값입니다. if (n <= 1) return n; 조건에서 n=0이면 0을 반환하고, n=1이면 1을 반환합니다. '1을 반환한다'고 착각하면 오답이 됩니다. 피보나치 수열의 첫 항은 0부터 시작(0,1,1,2,3,5,8,13…)함을 명심하세요.

순수 재귀는 중복 계산이 많아 지수 시간 복잡도 O(2^n)을 가집니다. 실무에서는 메모이제이션(Memoization)이나 동적 프로그래밍(DP)으로 O(n)으로 최적화합니다. 재귀 호출 스택을 손으로 직접 그려 추적하는 연습이 합격의 지름길입니다.

📌 재귀 함수(Recursive Function)와 피보나치 수열 완전 정리

재귀 함수(Recursive Function)는 함수가 자기 자신을 호출하는 프로그래밍 기법으로, 반드시 종료 조건(Base Case)이 있어야 무한 루프를 방지할 수 있습니다. 피보나치 수열(Fibonacci Sequence)은 F(0)=0, F(1)=1, F(n)=F(n-1)+F(n-2)로 정의되며 0, 1, 1, 2, 3, 5, 8, 13, 21… 순서로 증가합니다. 정보처리기사 실기 단골 함정은 f(0)=0이라는 점입니다. if(n <= 1) return n; 코드에서 n=0이면 0을 반환하므로, 시험 문제에서 f(0)을 1로 착각하여 틀리는 경우가 많습니다. 재귀 함수의 시간 복잡도는 단순 재귀 피보나치의 경우 O(2^n)이지만, 메모이제이션(Memoization)을 적용하면 O(n)으로 최적화됩니다. 재귀 함수는 스택(Stack) 자료구조를 활용하여 함수 호출 프레임을 관리하며, 과도한 재귀 호출은 스택 오버플로우(Stack Overflow)를 발생시킵니다.

  • 기저 조건(Base Case): 재귀 종료 조건 — 반드시 존재해야 함
  • 피보나치 수열: F(0)=0, F(1)=1이 핵심 — F(0)≠1 주의

Python 클로저와 Late Binding - 함정 완벽 해설

클로저(Closure)는 외부 함수의 변수를 기억하는 내부 함수(중첩 함수)입니다. Python 람다와 클로저는 변수를 값(Value)이 아닌 이름(Reference)으로 캡처합니다. 이를 Late Binding이라 하며, 변수의 값은 함수가 호출될 때 조회됩니다.

for 루프에서 lambda: i**2를 리스트에 담으면, 루프 변수 i는 이름 참조로 캡처됩니다. 루프가 끝난 후 i=2이므로, 모든 람다가 호출될 때 동일하게 2²=4를 반환합니다. 의도한 동작은 lambda i=i: i**2처럼 기본 인자로 값을 캡처하거나, 별도 팩토리 함수를 사용해야 합니다.

Python의 스코핑 규칙인 LEGB(Local→Enclosing→Global→Built-in) 순서로 이름을 조회합니다. 클로저에서 외부 변수를 수정하려면 nonlocal 키워드가 필요합니다. global은 전역 변수를, nonlocal은 바로 한 단계 외부 스코프의 변수를 수정합니다.

클로저(Closure)는 내부 함수가 외부 함수의 변수를 캡처(Capture)하여 외부 함수 종료 후에도 해당 변수에 접근할 수 있는 함수입니다. Python의 람다(lambda)와 중첩 함수는 클로저를 형성합니다.   • Early Binding 해결법: lambda i=i: i**2 — 기본값으로 현재 값 고정

윈도우 함수(Window Function)는 GROUP BY처럼 행을 그룹으로 묶지 않고, 각 행에 대해 집계 함수를 적용하는 SQL 고급 기능입니다. 함수명() OVER(PARTITION BY 컬럼 ORDER BY 컬럼) 형태로 사용합니다. SUM(SALARY) OVER()처럼 OVER() 안이 비어있으면 전체 결과셋을 대상으로 집계하므로 전체 합계를 각 행에 함께 표시할 수 있습니다. 이를 활용하면 각 행의 값이 전체 합계 대비 몇 퍼센트인지 한 번의 쿼리로 계산 가능합니다. 주요 윈도우 함수로는 순위 함수(RANK, DENSE_RANK, ROW_NUMBER), 집계 함수(SUM, AVG, COUNT), 값 함수(LAG, LEAD, FIRST_VALUE, LAST_VALUE)가 있습니다. PARTITION BY는 GROUP BY처럼 데이터를 그룹으로 나누지만 행을 축소시키지 않는 것이 핵심 차이입니다.

  • OVER(): 빈 괄호 → 전체 결과셋 대상
  • PARTITION BY: 그룹을 나누지만 행을 줄이지 않음
  • RANK vs DENSE_RANK: 공동 순위 다음 번호 건너뜀 여부 차이

DB · 트랜잭션 격리 수준

✓ ① REPEATABLE READ    ② SERIALIZABLE

격리 수준 4단계: READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE. Non-Repeatable Read를 막으려면 같은 트랜잭션 내 재조회 시 동일 결과를 보장하는 REPEATABLE READ 이상이 필요하다. Phantom Read까지 막으려면 최상위 격리 수준인 SERIALIZABLE이 필요하다.

트랜잭션 ACID 특성과 격리 수준(Isolation Level) 4단계

트랜잭션(Transaction)은 데이터베이스에서 하나의 논리적 작업 단위입니다. ACID는 원자성(Atomicity)·일관성(Consistency)·고립성(Isolation)·지속성(Durability)의 앞 글자를 딴 4가지 특성입니다. 원자성은 모든 연산이 전부 성공하거나 전부 실패해야 하며, ROLLBACK으로 실패 시 원래 상태로 복구합니다.

격리 수준(Isolation Level)은 동시 트랜잭션 간 데이터 접근을 제어합니다. 낮은 수준부터 READ UNCOMMITTED(Dirty Read 허용) → READ COMMITTED → REPEATABLE READ(Non-Repeatable Read 방지) → SERIALIZABLE(Phantom Read까지 방지) 순으로 엄격해집니다. 격리 수준이 높을수록 데이터 정합성은 높아지지만 동시성(처리량)은 낮아집니다.

발생 가능한 동시성 문제: Dirty Read(커밋 전 데이터 읽기), Non-Repeatable Read(같은 행을 두 번 읽을 때 값이 다름), Phantom Read(같은 조건 재조회 시 행 수가 달라짐). MySQL InnoDB의 기본 격리 수준은 REPEATABLE READ입니다.

📌 트랜잭션 격리 수준(Transaction Isolation Level) 4단계 완전 정리

트랜잭션(Transaction)의 ACID 특성 중 격리성(Isolation)을 제어하는 것이 격리 수준(Isolation Level)입니다. SQL 표준에서는 4단계를 정의합니다. READ UNCOMMITTED는 커밋되지 않은 데이터도 읽을 수 있어 Dirty Read 문제가 발생합니다. READ COMMITTED는 커밋된 데이터만 읽어 Dirty Read를 방지하지만 Non-Repeatable Read가 발생합니다. REPEATABLE READ는 트랜잭션 내 동일 조회 결과를 보장하지만 Phantom Read가 발생할 수 있습니다. MySQL InnoDB의 기본값입니다. SERIALIZABLE은 가장 엄격한 수준으로 Phantom Read까지 방지하지만 성능 저하가 큽니다. 격리 수준이 높을수록 동시성은 낮아지고 데이터 정합성은 높아지는 트레이드오프 관계입니다.

  • Dirty Read: 커밋 전 데이터 읽기 — READ UNCOMMITTED에서 발생
  • Non-Repeatable Read: 같은 행 재조회 시 값이 변경 — REPEATABLE READ로 방지
  • Phantom Read: 같은 조건 재조회 시 행 수 변경 — SERIALIZABLE로 방지

암기 Tip! "기순교절시논우" — 응집도는 높을수록(기능적에 가까울수록) 좋고, 결합도는 낮을수록 좋다.

응집도(Cohesion) 7단계와 결합도(Coupling) 6단계 - 핵심 암기법

응집도(Cohesion)는 모듈 내부 요소들이 얼마나 밀접하게 연관되어 있는지를 나타냅니다. 높은 응집도가 좋은 설계입니다. 7단계를 강한 것부터 약한 것 순으로 나열하면: 기능적(Functional) → 순차적(Sequential) → 교환적(Communicational) → 절차적(Procedural) → 시간적(Temporal) → 논리적(Logical) → 우연적(Coincidental)입니다. 암기법: "기순교절시논우"

📌 소프트웨어 모듈 응집도(Cohesion) 7단계 완전 암기법

응집도(Cohesion)는 모듈 내부 구성 요소들이 얼마나 밀접하게 관련되어 있는지를 나타내는 척도입니다. 높을수록 좋습니다. 7단계를 강한 것부터 약한 것으로 나열하면: 기능적(Functional) → 순차적(Sequential) → 통신적/교환적(Communicational) → 절차적(Procedural) → 시간적(Temporal) → 논리적(Logical) → 우연적(Coincidental) 입니다. 암기법: "기순교절시논우". 기능적 응집도는 단일 기능만 수행하는 이상적인 모듈이며, 우연적 응집도는 아무 관계 없는 기능이 한 모듈에 모인 최악의 경우입니다. 모듈화 설계에서 결합도(Coupling)는 낮게, 응집도(Cohesion)는 높게 유지해야 유지보수성과 재사용성이 향상됩니다. 응집도와 결합도는 소프트웨어 설계 품질 평가의 핵심 지표로 정보처리기사 실기에서 매우 자주 출제됩니다.

  • 우연적: 무관한 기능 집합 — 가장 나쁨

✓ 공통 결합도(Common Coupling)

결합도 강 → 약 순서: 내용 → 공통 → 외부 → 제어 → 스탬프 → 자료 (내·공·외·제·스·자). 전역 변수를 공유하는 것은 공통(Common) 결합도. 가장 강한 내용 결합도는 다른 모듈의 내부 코드를 직접 참조·수정하는 경우다.

소프트웨어 모듈 결합도(Coupling) - 공통 결합도(Common Coupling) 심화

결합도(Coupling)는 서로 다른 모듈 간의 상호의존성 정도입니다. 결합도가 높을수록 한 모듈의 변경이 다른 모듈에 파급 효과를 미쳐 유지보수가 어려워집니다. 공통 결합도(Common Coupling)는 두 모듈이 동일한 전역 변수(Global Variable)를 공유하는 경우입니다.

전역 변수는 어느 모듈에서도 변경 가능하므로 예측하기 어려운 부작용(Side Effect)이 발생할 수 있습니다. 이보다 강한 내용 결합도(Content Coupling)는 한 모듈이 다른 모듈의 내부 코드를 직접 참조하거나 수정하는 최악의 경우입니다. 반대로 가장 좋은 자료 결합도(Data Coupling)는 모듈 간에 단순 데이터(기본 타입)만 파라미터로 전달하는 경우입니다.

스탬프 결합도(Stamp Coupling)는 구조체·배열 등 복합 데이터를 전달하는 경우이며, 제어 결합도(Control Coupling)는 제어 신호(플래그)를 전달하여 상대 모듈의 흐름을 제어합니다. 소프트웨어 리팩토링 시 결합도를 낮추는 방향으로 설계를 개선해야 합니다.

결합도(Coupling)는 모듈과 모듈 사이의 의존 정도를 나타내며, 낮을수록 좋습니다. 강한 것부터 약한 것으로: 내용(Content) → 공통(Common) → 외부(External) → 제어(Control) → 스탬프(Stamp) → 자료(Data) 결합도입니다. 암기법: "내공외제스자". 내용 결합도는 한 모듈이 다른 모듈 내부 코드·데이터에 직접 접근하는 최악의 경우이고, 자료 결합도는 매개변수로 단순 데이터만 전달하는 가장 이상적인 형태입니다. 공통 결합도는 전역 변수를 여러 모듈이 공유하는 형태로, 변경 시 예상치 못한 부작용(Side Effect)이 발생합니다. 제어 결합도는 제어 신호(플래그)를 인수로 전달하여 다른 모듈의 흐름을 제어하는 것입니다. 높은 결합도는 변경 파급 효과(Ripple Effect)를 증대시켜 유지보수를 어렵게 합니다.

테스트 · 분기 커버리지

✓ 최소 3개 / (A>0, B>0), (A>0, B≤0), (A≤0, 임의)

분기(결정) 커버리지는 모든 결정 포인트의 True/False를 최소 1회씩 실행해야 한다. 분기점이 2개(A>0, B>0)이므로 필요 경로: ①A>0 & B>0, ②A>0 & B≤0, ③A≤0 → 총 3개 테스트 케이스. ※조건 커버리지(100%)와 혼동 주의.

소프트웨어 테스트 커버리지 - 구문·분기·조건·경로 커버리지 비교

화이트박스 테스트(White-Box Test)는 내부 코드 구조를 기반으로 테스트 케이스를 설계합니다. 커버리지(Coverage)는 테스트가 코드의 어느 부분을 얼마나 실행했는지 측정하는 지표입니다.

구문 커버리지(Statement Coverage)는 모든 문장을 최소 1회 실행, 분기 커버리지(Branch/Decision Coverage)는 모든 분기점의 참·거짓을 각각 1회 이상 실행, 조건 커버리지(Condition Coverage)는 각 조건식의 참·거짓을 1회 이상 실행, 경로 커버리지(Path Coverage)는 가능한 모든 실행 경로를 테스트합니다. 커버리지 강도: 경로 > MC/DC > 분기·조건 > 분기 > 구문 순입니다.

McCabe 사이클로매틱 복잡도(V(G))는 프로그램의 논리적 복잡도를 수치화합니다. V(G) = E(간선) - N(노드) + 2 또는 조건문(결정 포인트) 개수 + 1로 계산합니다. switch의 각 case는 별도 결정 포인트로 카운트됩니다. V(G) 값은 독립적 경로 수이자 최소 테스트 케이스 수를 의미합니다.

화이트박스 테스트(구조 기반 테스트)는 소스 코드 내부 구조를 기반으로 테스트 케이스를 설계합니다. 주요 커버리지(Coverage) 기준: 구문 커버리지(Statement)는 모든 구문을 1회 이상 실행. 분기 커버리지(Branch/Decision)는 모든 분기의 True/False를 1회씩 실행. 조건 커버리지(Condition)는 각 조건식의 True/False를 1회씩. 다중 조건 커버리지(MC/DC)는 각 조건이 독립적으로 결정에 영향. 경로 커버리지(Path)는 모든 실행 가능 경로를 실행. 분기 커버리지 100%가 구문 커버리지 100%를 포함하며, 커버리지 강도 순서는 경로 > 다중조건 > 조건/분기 > 구문입니다. McCabe의 사이클로매틱 복잡도 V(G) = 분기 조건 수 + 1로 최소 테스트 케이스 수를 산정할 수 있습니다.

  • V(G) 공식: 결정 포인트 수 + 1 = 최소 경로 수
  • 포함 관계: 경로 ⊃ MC/DC ⊃ 조건 ⊃ 분기 ⊃ 구문

📌 블랙박스 테스트(Black-Box Test) 기법 종류와 차이점

블랙박스 테스트(명세 기반 테스트)는 내부 구조를 모르고 입력·출력만으로 테스트합니다. 주요 기법: 동치 분할(Equivalence Partitioning)은 입력 도메인을 유효·무효 클래스로 분류하여 각 대표값으로 테스트합니다. 경계값 분석(BVA, Boundary Value Analysis)은 각 클래스의 경계(최솟값, 최솟값+1, 최댓값-1, 최댓값)에 집중하는 기법으로, 버그가 경계 근처에 집중되는 경험적 사실을 활용합니다. 원인-결과 그래프(Cause-Effect Graph)는 입력 조건 조합과 출력 결과의 관계를 그래프로 표현합니다. 오류 예측(Error Guessing)은 경험에 기반해 오류 발생 가능 입력값을 직관적으로 선정합니다. 페어와이즈(Pairwise) 테스트는 모든 파라미터 쌍의 조합을 최소한으로 커버합니다. 동치 분할로 클래스를 선정하고, 경계값 분석으로 각 클래스 경계를 테스트하는 조합이 실무·시험 모두에 자주 등장합니다.

  • 동치 분할: 클래스 대표값 선정 — 테스트 수 최소화
  • 경계값 분석: 경계 ±1 집중 — 실제 버그 발견율 높음
  • 오류 예측: 경험 기반 직관적 선정

McCabe 사이클로매틱 복잡도 계산법과 활용

사이클로매틱 복잡도(Cyclomatic Complexity)는 Thomas McCabe가 제안한 소프트웨어 복잡도 측정 지표입니다. 프로그램의 독립적 경로 수를 나타내며, V(G) = E - N + 2P 공식으로 계산합니다(E=간선 수, N=노드 수, P=연결된 컴포넌트 수, 단순 함수는 P=1).

실용적 계산법: 결정 포인트(조건문) 개수 + 1. if·while·for·case가 각각 1개, switch의 case는 case 개수만큼 추가됩니다. 일반적으로 V(G) ≤ 10이면 테스트하기 좋은 복잡도로, 10을 초과하면 리팩토링을 권고합니다. V(G) 값은 최소 테스트 케이스 수(분기 커버리지 100%)와 동일합니다.

정보처리기사 실기 코드 문제에서 복잡도를 계산할 때는 조건문을 모두 찾아 개수를 세고 1을 더하는 방법이 가장 빠릅니다. switch 문의 경우 case 수를 잊지 말고 세어야 합니다. default가 없는 switch도 각 case가 별도 결정 포인트입니다.

📌 McCabe 사이클로매틱 복잡도(Cyclomatic Complexity) 계산법

사이클로매틱 복잡도(Cyclomatic Complexity, CC)는 Thomas McCabe가 1976년 제안한 소프트웨어 복잡도 측정 지표입니다. 공식은 V(G) = E - N + 2P (E: 간선 수, N: 노드 수, P: 연결된 구성 요소 수)이며, 단일 프로그램에서는 P=1이므로 V(G) = 결정 포인트(조건문) 수 + 1로 간소화됩니다. if, while, for, case는 각각 1씩 추가하며, switch의 각 case도 별도 분기로 카운트합니다. V(G)는 독립 경로(Independent Path)의 수이자 최소 테스트 케이스 수를 의미합니다. V(G) ≤ 10이면 단순 모듈, 11~20이면 복잡, 21 이상이면 매우 복잡하여 리팩토링이 필요한 수준입니다. 정보처리기사 실기에서는 주어진 코드나 제어 흐름 그래프를 보고 V(G) 값을 계산하는 문제가 출제됩니다.

  • 공식 (간이): V(G) = 조건/분기문 수 + 1
  • switch 주의: 각 case가 별도 분기 → case 수만큼 가산
  • 권장 기준: V(G) ≤ 10 권장, 21 이상은 리팩토링 필요

SQL Injection과 구분해야 할 공격: XSS(스크립트 삽입→브라우저 실행), CSRF(사용자 인증 세션 악용), 커맨드 인젝션(OS 명령어 삽입). 2023~2025년 정보처리기사 실기에서 지속적으로 출제되는 핵심 보안 문제입니다.

✓ 대칭키: AES, DES, ARIA, 3-DES, SEED   |   비대칭키: RSA, ECC, DSA

대칭키(비밀키): 암·복호화에 같은 키 사용. 빠름. AES(128/192/256bit), DES(56bit, 취약), 3-DES, ARIA(국내 표준), SEED(국내 표준 → 64bit 블록). 비대칭키(공개키): 공개키/개인키 쌍. 느림. RSA(소인수분해), ECC(타원곡선), DSA(디지털서명 전용).

암호화 알고리즘 총정리 - 대칭키(AES·DES·ARIA·SEED) vs 비대칭키(RSA·ECC)

암호화는 크게 대칭키 암호화와 비대칭키(공개키) 암호화로 나뉩니다. 대칭키는 암호화·복호화에 동일한 키를 사용해 속도가 빠르지만, 키 배송 문제가 있습니다. AES(Advanced Encryption Standard)는 128/192/256비트 키를 지원하는 현재 표준, DES는 56비트로 취약해 사실상 폐기, 3-DES는 DES를 3회 적용, ARIA·SEED는 국내 표준 알고리즘입니다.

비대칭키는 공개키(Public Key)로 암호화하고 개인키(Private Key)로 복호화합니다. 키 쌍 생성 시 두 키가 수학적으로 연결됩니다. RSA는 큰 소수의 소인수분해 난이도를 기반으로 하며 현재 가장 널리 쓰입니다. ECC(타원곡선)는 RSA보다 짧은 키로 동등한 보안 제공, DSA는 디지털 서명 전용입니다.

해시 함수(Hash Function)도 보안의 핵심입니다. MD5(128비트, 취약), SHA-1(160비트, 취약), SHA-256/SHA-512(현재 표준)이 주요 알고리즘입니다. 해시는 단방향이므로 복호화가 불가능하며, 비밀번호 저장에 솔트(Salt)를 함께 사용합니다.

MD5(취약), SHA-1(취약), SHA-256/512(권장).

📌 C언어 포인터(Pointer) 기본 — 주소 연산자와 역참조

SQL GROUP BY·HAVING절 - WHERE와의 차이점과 집계 함수

SQL의 GROUP BY는 지정한 컬럼의 값이 동일한 행들을 하나의 그룹으로 묶어 집계 함수를 적용합니다. 주요 집계 함수: COUNT()·SUM()·AVG()·MAX()·MIN(). GROUP BY 절에 명시되지 않은 컬럼은 SELECT에서 집계 함수 없이 사용할 수 없습니다.

📌 SQL GROUP BY와 HAVING — 집계 함수와 조건 필터링

SQL에서 집계(Aggregation)는 여러 행을 그룹으로 묶어 하나의 결과를 반환하는 것입니다. GROUP BY는 지정한 컬럼을 기준으로 데이터를 그룹화하고, AVG(), SUM(), COUNT(), MAX(), MIN() 등 집계 함수를 적용합니다. 📌 SQL 명령어 분류 — DDL·DML·DCL·TCL 완전 정리

SQL(Structured Query Language)은 관계형 데이터베이스 관리 시스템(RDBMS)에서 데이터를 처리하기 위한 표준 언어입니다. 명령어 유형: DDL(Data Definition Language, 데이터 정의어)는 데이터베이스 객체(테이블, 뷰, 인덱스 등)를 생성·수정·삭제합니다. CREATE, ALTER, DROP, RENAME, TRUNCATE. DDL은 자동 커밋(Auto Commit)됩니다. DML(Data Manipulation Language, 데이터 조작어)는 테이블 내 데이터를 조작합니다. SELECT(조회), INSERT(삽입), UPDATE(수정), DELETE(삭제). DCL(Data Control Language, 데이터 제어어)은 데이터베이스 접근 권한을 제어합니다. GRANT(권한 부여), REVOKE(권한 회수). TCL(Transaction Control Language, 트랜잭션 제어어)은 DML 작업의 트랜잭션을 제어합니다. COMMIT(확정), ROLLBACK(취소), SAVEPOINT(저장점). TRUNCATE는 DDL이므로 ROLLBACK이 불가능하다는 점에서 DELETE(DML)와 다릅니다.

  • DDL: CREATE/ALTER/DROP — 자동 커밋, ROLLBACK 불가
  • TRUNCATE vs DELETE: DDL vs DML — 롤백 가능 여부 차이
  • TCL: COMMIT/ROLLBACK/SAVEPOINT — DML 트랜잭션 제어

📌 GoF 디자인 패턴 — 싱글톤(Singleton) 패턴 구현과 주의점

싱글톤(Singleton) 패턴은 GoF 생성(Creational) 패턴으로, 클래스의 인스턴스가 오직 하나만 존재하도록 보장하고 이 인스턴스에 대한 전역 접근점(Global Access Point)을 제공합니다. Java 구현: 생성자를 private으로 선언하여 외부에서 new를 막고, 클래스 내부에 static 멤버 변수로 인스턴스를 가지며, public static 메서드(getInstance())를 통해 접근합니다. 멀티스레드 환경에서는 Double-Checked Locking(DCL)이나 Enum 방식을 사용해야 스레드 안전성을 보장할 수 있습니다. Eager Initialization(클래스 로딩 시 즉시 생성)은 간단하지만 사용하지 않아도 자원을 소비합니다. Lazy Initialization(최초 접근 시 생성)은 자원 효율적이나 스레드 안전성 고려가 필요합니다. 싱글톤 패턴의 안티패턴(Anti-Pattern) 측면: 전역 상태를 가져 결합도를 높이고 테스트를 어렵게 합니다.

  • 구현 핵심: private 생성자 + static 인스턴스 + getInstance()
  • 멀티스레드: DCL 또는 Enum 방식으로 스레드 안전성 확보
  • 생성 패턴 5가지: Abstract Factory·Builder·Factory Method·Prototype·Singleton

SW개발모델 · V 모델

✓V 모델(V-Model)

V 모델에서 단계 대응 관계: 요구사항분석 ↔ 인수테스트, 시스템설계 ↔ 시스템테스트, 아키텍처설계 ↔ 통합테스트, 상세설계 ↔ 단위테스트. 각 개발 단계에서 테스트 계획을 동시에 수립하는 것이 특징이다.

V 모델(V-Model) - 개발 단계와 테스트 단계의 대응 관계

V 모델은 폭포수(Waterfall) 모델을 확장하여 개발 단계와 테스트 단계를 V자 형태로 대응시킨 소프트웨어 개발 프로세스입니다. 검증(Verification)—"올바르게 만들고 있는가?"와 확인(Validation)—"올바른 것을 만들고 있는가?"를 체계화합니다.

왼쪽(하강) 단계: 요구사항 분석 → 시스템 설계 → 아키텍처 설계 → 상세 설계 → 구현. 오른쪽(상승) 단계: 단위 테스트(상세설계 검증) → 통합 테스트(아키텍처 검증) → 시스템 테스트(시스템 설계 검증) → 인수 테스트(요구사항 검증). 각 개발 단계에서 해당 테스트 계획을 동시에 수립합니다.

V 모델의 단점은 요구사항 변경 시 대응이 어렵다는 점입니다. 이를 극복하기 위해 반복·점진적 개발 모델인 나선형(Spiral) 모델이나 애자일 방법론을 사용합니다. 정보처리기사 실기에서는 각 테스트 단계와 대응하는 개발 단계를 짝으로 암기해야 합니다.

V 모델검증 확인단위 테스트인수 테스트폭포수 모델
▶ 핵심 개념 보기 (+상세 해설)

📌 소프트웨어 개발 방법론 — V 모델(V-Model)과 폭포수·나선 비교

V 모델(V-Model)은 폭포수 모델(Waterfall Model)을 기반으로 각 개발 단계와 대응되는 테스트 단계를 V자 형태로 배치한 모델입니다. 왼쪽(개발) 하향: 요구사항 분석 → 시스템 설계 → 아키텍처 설계 → 상세 설계 → 코딩. 오른쪽(테스트) 상향: 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트. 각 개발 단계에서 대응하는 테스트 계획을 수립하는 것이 특징입니다. 폭포수 모델은 순차적 진행으로 이전 단계로 복귀가 어렵지만, 나선 모델(Spiral Model)은 위험 분석을 포함한 4단계(계획·위험분석·개발·평가)를 반복합니다. 애자일(Agile)은 짧은 스프린트로 점진적 개발하며 변화에 적응합니다. 프로토타입 모델은 시제품을 먼저 만들어 요구사항을 명확화합니다. 정보처리기사 실기에서는 각 모델의 특징과 대응 관계를 묻는 문제가 출제됩니다.

  • V 모델 대응: 요구분석↔인수테스트, 시스템설계↔시스템테스트, 상세설계↔단위테스트
  • 폭포수: 순차적, 되돌아가기 어려움
  • 나선 모델: 위험 분석 포함, 반복적 개발

기능적 요구사항: 시스템이 제공해야 할 서비스/기능 (예: "로그인 기능 제공"). 비기능적 요구사항: 시스템이 어떻게 동작해야 하는지의 품질 기준 (예: "응답시간 3초 이내", "99.9% 가용성"). 비기능적 요구사항은 제약사항이라고도 한다.

📌 소프트웨어 요구사항(Requirements) 분류와 명세 기법

소프트웨어 요구사항 분석은 SDLC(Software Development Life Cycle)에서 가장 중요한 초기 단계입니다. 요구사항은 크게 두 가지로 분류됩니다. 기능적 요구사항(Functional Requirements)은 시스템이 제공해야 할 기능·서비스를 기술합니다. "사용자가 로그인할 수 있어야 한다", "상품 검색 기능을 제공해야 한다"처럼 동사 형태로 표현됩니다. 비기능적 요구사항(Non-Functional Requirements)은 시스템의 품질 속성과 제약사항을 기술합니다. 성능(응답시간 3초 이내), 보안(SSL/TLS 암호화), 가용성(99.9% SLA), 유지보수성, 이식성 등이 해당합니다. 요구사항 명세 기법: 자연어(Natural Language), 구조적 명세(DFD, ERD), UML 다이어그램, 수학적 명세. 요구사항의 특성: 완전성(Complete), 일관성(Consistent), 검증 가능성(Verifiable), 추적 가능성(Traceable). 요구사항 관리 도구: JIRA, Confluence, IBM DOORS 등.

  • 기능적: 시스템이 "무엇을" 해야 하는지 — 기능 목록
  • 비기능적: 시스템이 "얼마나 잘" 해야 하는지 — 품질·제약
  • 요구사항 특성: 완전성·일관성·검증가능성·추적가능성

암기 Tip! Up(위로) → Driver(운전자=위에서 조종). Down(아래로) → Stub(하위 대역).

📌 통합 테스트(Integration Test) — 상향식·하향식·빅뱅 비교

통합 테스트(Integration Test)는 개별 단위 테스트를 마친 모듈들을 결합하여 인터페이스와 상호작용을 검증합니다. 주요 전략 세 가지입니다. 하향식(Top-Down) 통합은 최상위 모듈부터 하위 모듈로 테스트하며, 하위 모듈을 대신하는 스텁(Stub)이 필요합니다. 전체 시스템 흐름을 일찍 검증할 수 있는 장점이 있습니다. 상향식(Bottom-Up) 통합은 최하위 모듈부터 상위로 테스트하며, 상위 모듈 역할을 대신하는 드라이버(Driver)가 필요합니다. 하위 모듈의 재사용성이 높을 때 유리합니다. 빅뱅(Big Bang) 통합은 모든 모듈을 한꺼번에 통합하여 테스트합니다. 간단하지만 오류 발생 시 원인 파악이 어렵습니다. 회귀 테스트(Regression Test)는 변경 후 기존 기능이 정상 동작하는지 재검증합니다. CI/CD 파이프라인에서 자동화 통합 테스트가 핵심 역할을 합니다.

  • 하향식: 최상위→하위, Stub(하위 대역) 필요
  • 상향식: 최하위→상위, Driver(상위 대역) 필요
  • 암기: Top-Down→Stub(아래에서 받침), Bottom-Up→Driver(위에서 운전)

📌 소프트웨어 유지보수성(Maintainability) — ISO 25010 하위 특성

소프트웨어 유지보수(Software Maintenance)는 배포 후 발생하는 변경 활동 전체를 의미합니다. 4가지 유지보수 유형: ① 수정(Corrective): 발견된 결함·오류 수정. ② 예방(Preventive): 미래 문제 방지를 위한 선제적 수정. ③ 적응(Adaptive): 환경 변화(OS 업그레이드, 법규 변경)에 대응. ④ 완전화(Perfective): 성능 개선·기능 추가·사용성 향상. ISO/IEC 25010의 유지보수성(Maintainability) 하위 5가지 특성: 모듈성(Modularity), 재사용성(Reusability), 분석성(Analysability, 진단 용이), 변경성(Changeability, 변경 용이), 시험성(Testability, 테스트 용이). 높은 유지보수성을 위해서는 낮은 결합도, 높은 응집도, 충분한 문서화, 코딩 컨벤션 준수, 단위 테스트 커버리지 유지가 중요합니다.

  • 4가지 유지보수: 수정·예방·적응·완전화
  • 분석성: 결함 진단·원인 파악이 용이한 정도
  • 변경성: 변경 요청에 적은 노력으로 대응 가능한 정도

📌 리팩토링(Refactoring)과 기술 부채(Technical Debt) 관리

리팩토링(Refactoring)은 마틴 파울러(Martin Fowler)가 체계화한 개념으로, 외부 동작(기능)을 변경하지 않고 내부 코드 구조를 개선하는 작업입니다. 코드 냄새(Code Smell) — 코드 품질 문제의 징후 — 를 식별하고 제거하는 것이 목적입니다. 주요 리팩토링 기법: 메서드 추출(Extract Method), 메서드 인라인(Inline Method), 조건문 분해(Decompose Conditional), 임시변수 제거(Replace Temp with Query), 매직 넘버를 상수로 변환(Replace Magic Number), 서브클래스 추출(Extract Subclass). 리팩토링은 반드시 단위 테스트 기반으로 수행해야 기능 변경이 없음을 보장합니다. 기술 부채(Technical Debt)는 빠른 개발을 위해 최적화되지 않은 코드를 사용한 결과로 쌓이는 미래 유지보수 비용입니다. 리팩토링은 기술 부채를 줄이고 코드베이스의 지속 가능성을 높이는 핵심 활동입니다.

  • 핵심 원칙: 외부 동작 불변 + 내부 구조 개선
  • 전제 조건: 충분한 단위 테스트 커버리지 확보
  • 기술 부채: 지름길 코드가 미래에 지불할 유지보수 비용

📌 XSS(Cross-Site Scripting) 공격 — 유형과 방어 기법

OS · SJF 평균 대기 시간

✓평균 대기 시간: 4 ms

SJF(비선점)는 실행 시간이 짧은 순서로 처리한다. 정렬: P3(2) → P2(4) → P4(6) → P1(8)
P3 대기: 0ms | P2 대기: 2ms | P4 대기: 6ms | P1 대기: 12ms
평균 대기 = (0 + 2 + 6 + 12) / 4 = 20 / 4 = 4ms.
(※ FCFS라면 평균 대기 = (0+8+12+15)/4 = 8.75ms로 SJF보다 훨씬 큼)

CPU 스케줄링 알고리즘 - SJF·FCFS·RR·우선순위 스케줄링 비교

CPU 스케줄링은 준비 큐(Ready Queue)에 있는 프로세스 중 어떤 것을 CPU에 할당할지 결정하는 운영체제의 핵심 기능입니다. 주요 알고리즘: FCFS(First Come First Served)—도착 순서대로 처리, SJF(Shortest Job First)—실행 시간이 짧은 것 우선, RR(Round Robin)—타임 슬라이스씩 순환, Priority—우선순위 순.

SJF(비선점)는 평균 대기 시간이 최소화되는 최적 알고리즘이지만, 실행 시간을 미리 알 수 없어 실제 적용이 어렵고 기아(Starvation) 문제가 발생합니다. 평균 대기 시간 = (각 프로세스의 대기 시간 합) / 프로세스 수. 대기 시간 = 완료 시간 - 도착 시간 - 실행 시간.

선점형(Preemptive) vs 비선점형(Non-Preemptive): 선점형은 실행 중인 프로세스를 중단할 수 있고(SRTF·RR·Priority 선점형), 비선점형은 프로세스가 자발적으로 CPU를 반납합니다(FCFS·SJF 비선점형). 정보처리기사 실기에서는 간트 차트를 그려 평균 대기 시간·반환 시간을 계산하는 문제가 자주 출제됩니다.

📌 CPU 스케줄링 — SJF·FCFS·Round Robin·Priority 비교

CPU 스케줄링은 운영체제가 프로세스들에게 CPU를 할당하는 정책입니다. 주요 알고리즘: FCFS(First-Come First-Served)는 도착 순서대로 처리하는 가장 단순한 방식으로, 비선점형이며 호위 효과(Convoy Effect) 문제가 있습니다. SJF(Shortest Job First)는 실행 시간이 짧은 프로세스 우선 처리합니다. 평균 대기 시간이 최소화되는 최적 알고리즘이나, 긴 작업의 기아(Starvation) 문제가 있습니다. 비선점형과 선점형(SRTF, Shortest Remaining Time First) 버전이 있습니다. 라운드 로빈(Round Robin)은 각 프로세스에 동일한 시간 할당량(Time Quantum)을 부여하며, 시간 할당량이 너무 작으면 컨텍스트 스위칭 오버헤드가 커집니다. 우선순위(Priority)는 우선순위가 높은 프로세스 먼저 처리하며, 에이징(Aging) 기법으로 기아 문제를 완화합니다. SJF 비선점의 평균 대기 시간 = 각 프로세스 대기 시간의 평균값입니다.

  • SJF 평균 대기: 짧은 순서 정렬 후 각 대기 시간 평균
  • FCFS 문제: Convoy Effect — 긴 작업이 짧은 작업을 막음
  • Round Robin: 타임 퀀텀 크기가 성능 좌우

2NF: 완전 함수 종속 — 부분 함수 종속 제거.

문제의 "완전 함수 종속" "부분 함수 종속 제거" 키워드 → 2NF.

데이터베이스 정규화(Normalization) 1NF~BCNF 완전 정복

정규화(Normalization)는 데이터 중복을 제거하고 이상현상(Anomaly)을 방지하기 위해 테이블을 체계적으로 분해하는 과정입니다. 삽입 이상·삭제 이상·갱신 이상을 해결합니다.

1NF(제1정규형): 모든 속성이 원자값(Atomic Value)—더 이상 분해할 수 없는 값. 2NF: 1NF + 부분 함수 종속 제거(기본키의 일부에만 종속되는 속성 분리). 복합 기본키에서 주로 발생. 3NF: 2NF + 이행적 함수 종속 제거(A→B→C 관계에서 A→C 종속 제거). BCNF(Boyce-Codd NF): 모든 결정자가 후보키. 3NF보다 강한 조건.

함수 종속(Functional Dependency) F.D.: X→Y는 X의 값이 Y의 값을 유일하게 결정함을 의미합니다. 후보키(Candidate Key)는 튜플을 유일하게 식별하는 속성의 최소 집합이며, 그중 하나를 기본키(Primary Key)로 선택합니다. 역정규화(Denormalization)는 성능을 위해 의도적으로 중복을 허용하는 것입니다.

📌 데이터베이스 정규화(Normalization) — 1NF부터 BCNF까지

정규화(Normalization)는 관계형 데이터베이스에서 데이터 중복을 최소화하고 이상 현상(Anomaly)을 방지하기 위해 테이블을 분해하는 과정입니다. 이상 현상 3가지: 삽입 이상(불필요한 데이터 삽입 강제), 삭제 이상(의도치 않은 데이터 삭제), 수정 이상(데이터 불일치 발생). 정규형 단계: 1NF(제1정규형): 모든 속성 값이 원자값(Atomic Value)이어야 함. 반복 그룹 제거. 2NF(제2정규형): 1NF + 완전 함수 종속. 복합 기본키에서 일부에만 종속된 속성(부분 함수 종속) 제거. 3NF(제3정규형): 2NF + 이행적 함수 종속 제거. A→B, B→C이면 A→C인 이행 종속 제거. BCNF(Boyce-Codd NF): 모든 결정자(Determinant)가 후보키(Candidate Key)여야 함. 3NF보다 엄격합니다. 정규화 단계가 높아질수록 중복은 줄지만 조인 연산이 증가할 수 있어 성능과의 트레이드오프를 고려합니다.

  • 2NF: 부분 함수 종속 제거 — 복합키 일부에만 종속된 속성 분리
  • 3NF: 이행적 함수 종속 제거 — A→B→C 형태 분리
  • BCNF: 모든 결정자가 후보키 — 가장 엄격한 정규형

📌 GoF 디자인 패턴 — 옵저버(Observer) 패턴과 Publish-Subscribe 모델

옵저버(Observer) 패턴은 GoF 행위(Behavioral) 패턴으로, 객체의 상태 변화를 관찰하는 관찰자들에게 자동으로 통보하는 1:N 의존 관계를 정의합니다. 구성 요소: Subject(주제)는 관찰 대상으로 Observer 목록을 관리하고 상태 변화 시 notify()를 호출합니다. Observer(관찰자)는 update() 메서드를 구현하여 알림을 받습니다. ConcreteSubject와 ConcreteObserver는 각각의 구체적 구현입니다. 장점: 느슨한 결합(Loose Coupling) — Subject는 Observer 인터페이스만 알면 됩니다. MVC 패턴의 Model-View 관계, GUI 이벤트 리스너(Java EventListener), Android LiveData, JavaScript의 addEventListener 등이 모두 옵저버 패턴을 기반으로 합니다. Publish-Subscribe(Pub-Sub) 패턴은 옵저버의 변형으로, 발행자와 구독자 사이에 메시지 브로커(Broker)를 두어 더욱 느슨한 결합을 달성합니다. Apache Kafka, RabbitMQ가 대표적인 Pub-Sub 시스템입니다.

  • 핵심: Subject 상태 변화 → 모든 Observer 자동 알림
  • 실제 예시: Java EventListener, Android LiveData, JavaScript addEventListener
  • Pub-Sub 차이: 중간 브로커(Broker) 존재 — 더 느슨한 결합
